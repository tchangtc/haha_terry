# Terry Skills 技能系统

[English](./README.md) | [简体中文](./README_zh-CN.md) | [繁體中文](./README_zh-TW.md)

Skills 是可复用的指令集，教 Terry 按特定领域的最佳实践执行任务。可以理解为"上岗培训手册"——当技能激活时，Terry 会自动遵循领域规范，无需你每次都解释步骤。

---

## 快速上手

想创建一个技能？这是最小示例：

```bash
mkdir -p ~/.terry/skills/my-reviewer

cat > ~/.terry/skills/my-reviewer/SKILL.md << 'EOF'
---
name: my-reviewer
description: 提交前的快速代码检查
triggers:
  - pre-commit check
  - 提交前检查
---

# 提交前审查

## 流程

1. 用 `git diff --name-only` 查看变更文件
2. 对每个文件检查：
   - 是否有调试用的 print 或 console.log
   - 是否有硬编码的密钥或密码
   - 是否缺少类型标注或文档字符串
3. 报告发现的问题。如果没问题，回复"可以提交"。

## 工具

- `git_diff` — 查看变更内容
- `read_file` — 检查变更的文件
- `grep` — 搜索调试语句
EOF

# 重新加载并激活
/reload-skills
提交前检查
```

就这么简单。一个文件夹、一个 `SKILL.md` 文件、一段 YAML 配置、再加 Markdown 格式的指令。

## 工作原理

```
用户输入消息
        │
        ▼
Terry 扫描所有技能的触发词
        │
   ┌────┴────┐
   ▼         ▼
 匹配到？   未匹配？
   │         │
   ▼         ▼
注入技能    正常代理
上下文      行为
```

当触发词匹配时，Terry 将技能指令注入到系统提示中。LLM 将这些指令视为该任务的"标准作业流程"。

**关键点：** 技能不替代工具——它指导工具*如何使用*。`code-review` 技能不会新增工具，而是告诉 Terry *用哪些*工具、*按什么顺序*、*关注什么*。

---

## SKILL.md 格式

```markdown
---
name: my-skill           # 必需：唯一标识（小写、连字符）
description: 功能描述      # 必需：一句话简介
triggers:                # 必需：激活此技能的触发短语
  - 触发短语
  - trigger phrase
---

# 技能标题

指令内容——任何有效的 Markdown 都可以。

## 流程

代理应遵循的分步骤工作流。

## 工具

代理应使用的工具列表，附简要理由。

## 指南

此工作流的最佳实践和约束条件。
```

### 触发词匹配

Terry 按以下顺序匹配触发词：

1. **精确子串** — 用户消息中包含触发短语
2. **名称匹配** — 用户消息中包含技能名称
3. **描述匹配** — 用户消息与技能描述有重叠

示例：输入 `"帮我审查这段代码"` 会匹配到 `code-review` 技能，因为 `"审查代码"` 是其触发词之一。

## 内置技能

Terry 预装了三个技能：

### `code-review`（代码审查）

系统化代码审查，涵盖安全性、性能、可维护性和最佳实践。生成按严重程度排序的结构化报告。

**触发词：** `code review`、`review code`、`代码审查`、`审查代码`

### `data-analysis`（数据分析）

分析数据文件（CSV、JSON、日志）。执行统计分析、趋势检测，并建议可视化方案。

**触发词：** `data analysis`、`analyze data`、`数据分析`、`分析数据`

### `document-generator`（文档生成）

生成专业文档——技术报告、项目提案、API 文档、用户指南——具备规范的结构和格式。

**触发词：** `generate document`、`create report`、`生成文档`、`创建报告`

## 创建优质技能

### 1. 写好触发词

**好的** — 具体、无歧义：
```yaml
triggers:
  - code review
  - review my code
  - 代码审查
```

**避免** — 太泛，会频繁误触发：
```yaml
triggers:
  - help
  - check
  - 帮我
```

### 2. 结构化流程

给代理清晰的执行顺序：

```markdown
## 流程

1. **理解需求** — 读取相关文件
2. **分析问题** — 识别问题或模式
3. **执行操作** — 应用修复或生成输出
4. **验证结果** — 确认输出正确
```

### 3. 明确工具用法

不要只列工具名——解释*何时*和*为什么*使用：

```markdown
## 工具

- `read_file` — 先读取目标文件，理解其结构
- `grep` — 搜索 TODO、FIXME 和硬编码值
- `edit_file` — 逐个应用修复，附带清晰的提交信息
```

### 4. 提供示例

具体示例帮助 LLM 理解预期行为：

```markdown
## 示例

**输入：** "审查 auth.py"

**输出：**
1. 发现：第 45 行存在 SQL 注入风险
2. 警告：登录接口缺少速率限制
3. 建议：将 JWT 逻辑提取到独立模块
```

## 技能管理

```bash
/skills              # 列出所有已安装技能
/skill <名称>        # 查看技能详情和内容
/activate <名称>     # 手动激活技能
/deactivate          # 停用当前技能
/reload-skills       # 从磁盘重新加载全部技能
/auto-skills         # 查看自动生成的技能（从对话模式中学习）
/auto-skill-approve  # 将自动生成的技能升级为永久技能
```

---

## 自动创建技能

Terry 会观察你在对话中的重复工作流。多次检测到同一模式后，会提议创建可复用技能：

```
你：  "找出项目里所有 SQL 查询并检查注入风险"
Terry：[搜索、发现问题、给出报告]

... 几轮对话后 ...

你：  "审查新 API 接口的安全性"
Terry：[搜索、发现问题、给出报告]

Terry："我注意到你经常做安全审计，我为你草拟了一个技能——用 /auto-skills 查看"
```

这样，Terry 的技能库会随着你的使用习惯不断增长。

---

## 技能目录

Terry 从以下位置加载技能（按优先级）：

1. `./skills/` — 项目专属技能（随仓库提交）
2. `~/.terry/skills/` — 个人技能（跨项目可用）

## 故障排除

**技能未出现在 `/skills` 中**
- 检查 `SKILL.md` 是否有有效的 YAML frontmatter（用 `---` 包裹）
- 确认 `name`、`description`、`triggers` 三个字段都存在
- 执行 `/reload-skills`

**技能不会自动激活**
- 确认触发短语确实出现在用户消息中
- 添加更多触发词变体（中英文都加上）
- 用 `/activate <名称>` 手动测试

**技能输出不符合预期**
- 检查技能指令是否有歧义
- 添加更具体的示例引导 LLM
- 检查流程步骤——顺序对吗？

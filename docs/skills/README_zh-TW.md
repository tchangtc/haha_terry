# Terry Skills 技能系統

[English](./README.md) | [简体中文](./README_zh-CN.md) | [繁體中文](./README_zh-TW.md)

Skills 是可複用的指令集，教 Terry 按特定領域的最佳實踐執行任務。可以理解為「上崗培訓手冊」——當技能啟用時，Terry 會自動遵循領域規範，無需你每次都解釋步驟。

---

## 快速上手

想建立一個技能？這是最小範例：

```bash
mkdir -p ~/.terry/skills/my-reviewer

cat > ~/.terry/skills/my-reviewer/SKILL.md << 'EOF'
---
name: my-reviewer
description: 提交前的快速程式碼檢查
triggers:
  - pre-commit check
  - 提交前檢查
---

# 提交前審查

## 流程

1. 用 `git diff --name-only` 檢視變更檔案
2. 對每個檔案檢查：
   - 是否有除錯用的 print 或 console.log
   - 是否有硬編碼的金鑰或密碼
   - 是否缺少型別標註或文件字串
3. 回報發現的問題。如果沒問題，回覆「可以提交」。

## 工具

- `git_diff` — 檢視變更內容
- `read_file` — 檢查變更的檔案
- `grep` — 搜尋除錯語句
EOF

# 重新載入並啟用
/reload-skills
提交前檢查
```

就這麼簡單。一個資料夾、一個 `SKILL.md` 檔案、一段 YAML 設定、再加 Markdown 格式的指令。

---

## 工作原理

```
使用者輸入訊息
        │
        ▼
Terry 掃描所有技能的觸發詞
        │
   ┌────┴────┐
   ▼         ▼
 匹配到？   未匹配？
   │         │
   ▼         ▼
注入技能    正常代理
上下文      行為
```

當觸發詞匹配時，Terry 將技能指令注入到系統提示中。LLM 將這些指令視為該任務的「標準作業流程」。

**關鍵點：** 技能不替代工具——它指導工具*如何使用*。`code-review` 技能不會新增工具，而是告訴 Terry *用哪些*工具、*按什麼順序*、*關注什麼*。

---

## SKILL.md 格式

```markdown
---
name: my-skill           # 必需：唯一識別碼（小寫、連字號）
description: 功能描述      # 必需：一句話簡介
triggers:                # 必需：啟用此技能的觸發短語
  - 觸發短語
  - trigger phrase
---

# 技能標題

指令內容——任何有效的 Markdown 都可以。

## 流程

代理應遵循的分步驟工作流。

## 工具

代理應使用的工具列表，附簡要理由。

## 指南

此工作流的最佳實踐和約束條件。
```

### 觸發詞匹配

Terry 按以下順序匹配觸發詞：

1. **精確子字串** — 使用者訊息中包含觸發短語
2. **名稱匹配** — 使用者訊息中包含技能名稱
3. **描述匹配** — 使用者訊息與技能描述有重疊

範例：輸入 `「幫我審查這段程式碼」` 會匹配到 `code-review` 技能，因為 `「審查程式碼」` 是其觸發詞之一。

## 內建技能

Terry 預裝了三個技能：

### `code-review`（程式碼審查）

系統化程式碼審查，涵蓋安全性、效能、可維護性和最佳實踐。生成按嚴重程度排序的結構化報告。

**觸發詞：** `code review`、`review code`、`程式碼審查`、`審查程式碼`

### `data-analysis`（資料分析）

分析資料檔案（CSV、JSON、日誌）。執行統計分析、趨勢檢測，並建議視覺化方案。

**觸發詞：** `data analysis`、`analyze data`、`資料分析`、`分析資料`

### `document-generator`（文件生成）

生成專業文件——技術報告、專案提案、API 文件、使用者指南——具備規範的結構和格式。

**觸發詞：** `generate document`、`create report`、`生成文件`、`建立報告`

## 建立優質技能

### 1. 寫好觸發詞

**好的** — 具體、無歧義：
```yaml
triggers:
  - code review
  - review my code
  - 程式碼審查
```

**避免** — 太泛，會頻繁誤觸發：
```yaml
triggers:
  - help
  - check
  - 幫我
```

### 2. 結構化流程

給代理清晰的執行順序：

```markdown
## 流程

1. **理解需求** — 讀取相關檔案
2. **分析問題** — 識別問題或模式
3. **執行操作** — 套用修復或生成輸出
4. **驗證結果** — 確認輸出正確
```

### 3. 明確工具用法

不要只列工具名——解釋*何時*和*為什麼*使用：

```markdown
## 工具

- `read_file` — 先讀取目標檔案，理解其結構
- `grep` — 搜尋 TODO、FIXME 和硬編碼值
- `edit_file` — 逐個套用修復，附帶清晰的提交訊息
```

### 4. 提供範例

具體範例幫助 LLM 理解預期行為：

```markdown
## 範例

**輸入：** "審查 auth.py"

**輸出：**
1. 發現：第 45 行存在 SQL 注入風險
2. 警告：登入端點缺少速率限制
3. 建議：將 JWT 邏輯提取到獨立模組
```

## 技能管理

```bash
/skills              # 列出所有已安裝技能
/skill <名稱>        # 檢視技能詳情和內容
/activate <名稱>     # 手動啟用技能
/deactivate          # 停用當前技能
/reload-skills       # 從磁碟重新載入全部技能
/auto-skills         # 檢視自動生成的技能（從對話模式中學習）
/auto-skill-approve  # 將自動生成的技能升級為永久技能
```

---

## 自動建立技能

Terry 會觀察你在對話中的重複工作流。多次偵測到同一模式後，會提議建立可複用技能：

```
你：  "找出專案裡所有 SQL 查詢並檢查注入風險"
Terry：[搜尋、發現問題、給出報告]

... 幾輪對話後 ...

你：  "審查新 API 端點的安全性"
Terry：[搜尋、發現問題、給出報告]

Terry："我注意到你經常做安全審計，我為你草擬了一個技能——用 /auto-skills 檢視"
```

這樣，Terry 的技能庫會隨著你的使用習慣不斷成長。

---

## 技能目錄

Terry 從以下位置載入技能（按優先級）：

1. `./skills/` — 專案專屬技能（隨倉庫提交）
2. `~/.terry/skills/` — 個人技能（跨專案可用）

## 故障排除

**技能未出現在 `/skills` 中**
- 檢查 `SKILL.md` 是否有有效的 YAML frontmatter（用 `---` 包裹）
- 確認 `name`、`description`、`triggers` 三個欄位都存在
- 執行 `/reload-skills`

**技能不會自動啟用**
- 確認觸發短語確實出現在使用者訊息中
- 新增更多觸發詞變體（中英文都加上）
- 用 `/activate <名稱>` 手動測試

**技能輸出不符合預期**
- 檢查技能指令是否有歧義
- 新增更具體的範例引導 LLM
- 檢查流程步驟——順序對嗎？
